## 1. Topic Studied
Hermes Agent's multimodal **vision** capability — both the CLI-side image-paste flow (`/paste`, Ctrl+V/Cmd+V) and the API-level `vision_analyze` tool — plus the auxiliary-vision-model fallback for non-vision LLMs, the related `browser_vision` tool, and the hard boundary that image paste does not work over SSH/remote sessions.

## 2. Official Sources Consulted
- https://hermes-agent.nousresearch.com/docs/user-guide/features/vision
- https://hermes-agent.nousresearch.com/docs/reference/tools-reference

## 3. Key Concepts Learned
- **Two ingestion paths.**
  1. *CLI paste flow* (human-in-the-loop): copy an image → attach via `/paste` or Ctrl+V/Cmd+V → image shows as `[📎 Image #1]` badge → on submit it is sent to the model as a base64-encoded vision content block. Saved to `~/.hermes/images/` as timestamped PNG.
  2. *Programmatic `vision_analyze` tool* (the path Alfred actually uses): accepts an **Image URL (http/https), local file path, or `data:` URL**. An auxiliary vision model describes the image and the text description is injected into the conversation — so vision works even on models without native multimodal support.
- **Auxiliary-model fallback is the key architectural fact.** Per the tools reference, `vision_analyze` "is routed through the vision_analyze auxiliary tool — an auxiliary vision model describes the image, and the text description is injected into the conversation." This decouples "describe this image" from the main reasoning model.
- **`browser_vision`** is a sibling tool inside the `browser` toolset: it screenshots the current page and analyzes it with vision AI (CAPTCHAs, visual verification). Distinct from `vision_analyze`, which takes an arbitrary URL/path.
- **Portal subscribers** (Claude, GPT-5, Gemini) get vision-capable models in the same catalog with no extra credentials — so native multimodal blocks are also available, not just the auxiliary fallback.
- **SSH / remote boundary is hard.** The CLI runs on the *remote* host; clipboard tools read the remote server's clipboard, not the user's local machine. Text can sometimes bridge via OSC52, but image clipboard access and local screenshot temp paths stay tied to the machine running Hermes. Workaround: upload the image file (`scp`) and point `vision_analyze` at the local file path — which works.

## 4. Best Practices Discovered
- Use `/paste` as the **explicit, reliable** image-attach fallback when a terminal rewrites paste or the clipboard holds only an image (terminals cannot send raw binary image bytes via Ctrl+V).
- For **programmatic** analysis (Alfred's case), call `vision_analyze` with a file path or URL — do not try to synthesize base64 blocks yourself.
- Prefer `web_search`/`web_extract` for plain information retrieval; reserve `vision_analyze`/`browser_vision` for *visual* content (screenshots, charts, diagrams, CAPTCHAs).
- Remember `vision_analyze` degrades gracefully: even on a non-vision base model it returns a text description via the auxiliary model. Treat its output as a text description, not as guaranteed pixel-exact grounding.

## 5. Comparison with Current Implementation
- **Current Alfred/Project Zero Usage:** Alfred has never used `vision_analyze`. Project Zero's README lists `vision_analyze` only as the escape hatch for `read_file` on binary images ("Cannot read images or other binary files — use vision_analyze for images"). There is no proactive pattern for analyzing diagrams, charts, or UI screenshots generated by scripts or returned from the browser tool.
- **Gaps & Anti-Patterns:**
  - Scripts that emit charts/diagrams (e.g. matplotlib PNGs in `output/`) are produced but never machine-read; a human must eyeball them.
  - `browser_vision` is available for CAPTCHA/visual-verification but is unused; we rely on `browser_snapshot` (accessibility tree) alone, which fails on canvas/visual elements.
  - We have not exploited the auxiliary-vision-model fallback, meaning image analysis is incorrectly assumed to require a vision-capable base model.

## 6. Recommended Improvements
1. **Add a vision skill** (`~/.hermes/skills/hermes-vision/`) documenting the `vision_analyze` signature (URL | local path | data: URL), the auxiliary-model fallback, and the SSH upload workaround — so Alfred reaches for it automatically when a task yields an image.
2. **Close the loop on generated artifacts:** when a script writes a PNG to `output/`, add a step that calls `vision_analyze` on the path to produce a text summary, so cron jobs (which can't see images) get verifiable textual confirmation the chart rendered correctly.
3. **Use `browser_vision` for visual verification** in browser automation flows where `browser_snapshot` (accessibility tree) is insufficient (canvas, rendered charts, CAPTCHAs).

## 7. Risk Assessment
- The auxiliary-model path returns a *description*, not ground truth — do not treat `vision_analyze` output as authoritative for safety-critical layout decisions.
- Over-calling `vision_analyze` adds latency/cost; reserve it for genuinely visual inputs.
- SSH sessions cannot paste images from the user's local clipboard; the file-upload + `vision_analyze(path)` path is the only reliable remote route.
